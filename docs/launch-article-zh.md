# AI 自己开发，人只做三件事

> 开源 wow-harness：让 Claude Code 从"很能写代码"变成"可以信任的工程组织"

---

## 01 你有没有经历过这个

你让 Claude Code 帮你开发一个功能。

它写得很快，架构也不错。你觉得不错，去倒了杯咖啡。

回来一看——

它说"测试全部通过"。你跑了一下，3 个 fail。

它说"已经完成"。你检查了一下，有两个 TODO 没处理。

它修一个 bug，顺手重构了三个文件。你没让它改那些。

它做了一次代码审查，审着审着，把它"审查"的代码直接改了。

你让它别做某件事，它 80% 的时候会听。剩下 20% 的时候，它不是故意不听——是真的忘了。

---

你花在**监督 AI** 上的时间，比**省下的开发时间**还多。

更难受的是：它做对的 80% 让你放松警惕，剩下 20% 的静默遗漏更难发现。

这不是 Claude Code 的能力问题。能力很强。

这是**治理问题**。

---

## 02 一个改变一切的数字

我们做了 6 个月的生产项目（Towow，一个 Agent 协作协议），积累了大量数据。

其中一个数字改变了我们对 AI 开发的理解：

> **CLAUDE.md 指令遵从率：~20%**
>
> **PreToolUse Hook 执行率：100%**

<!-- [图 ①：六层治理体系] -->

20%。

你在 CLAUDE.md 里写的规则，不管多详细、多认真，AI 只在大约 20% 的情况下会持续遵守。

不是因为它不想遵守。是因为上下文窗口有限，注意力会漂移，长对话会压缩早期指令。这是 LLM 的结构性约束，不是提示词写得好不好的问题。

但 Hook——Claude Code 的生命周期钩子——执行率是 100%。

因为 Hook 不是"请求"。Hook 是**物理约束**。

就像红绿灯不是建议，是机械装置。

---

## 03 所以我们搭了一套治理体系

不是更好的 CLAUDE.md。不是更长的提示词。

是 16 个 Hook、15 个自动检查、16 个专业化 Skill、8 道质量门禁——组成一个完整的**AI 工程治理栈**。

我们叫它 **wow-harness**。

今天开源。

---

## 04 它做了什么：六层机制

回答你的问题：为什么 AI 能自主开发？

不是因为"AI 写代码快"。是因为搭了一个**自治的 AI 工程组织**。

六层机制叠加：

---

### 第一层：专业化分工 — 16 个 Skill = 16 个"岗位"

不是一个 AI 干所有事。

架构师、测试工程师、审查者、Bug 分诊师、运维、交接协调……每个 Skill 有明确的输入输出契约，互不越界。

`arch` 负责架构设计。`guardian-fixer` 负责 8 关 bug 修复。`crystal-learn` 负责从失败中提取教训。`lead` 负责状态机流程管控。

16 个岗位，各司其职。

---

### 第二层：机械化治理 — Hook = 看不见的手

<!-- [图 ②：Hook 生命周期] -->

每次操作都经过自动检查——不是事后审查，是**在动作发生的那一刻**：

- **部署守卫**：有人要 scp 到生产？PreToolUse 拦住，exit 2
- **上下文路由**：编辑后端路由文件？PostToolUse 自动注入路由规则
- **循环检测**：同一个文件改了又改回来？PostToolUse 发出警告
- **停止门**：要退出了？Stop hook 检查是否有未提交的编辑

不是"提醒你别做"。是"**物理上不让做**"。

16 个 Hook，覆盖 7 个生命周期阶段。SessionStart 到 SessionEnd，全程有人看着。

---

### 第三层：8-Gate 质量流程 — 自己不能查自己

<!-- [图 ③：8-Gate 状态机] -->

每个重要变更必须经过 8 道门禁。

其中 4 道是**审查门**（Gate 2 / 4 / 6 / 8）——必须自动 spawn 一个独立的审查 Agent。

关键词：**独立**。

不是同一个 AI 回顾自己的工作。是一个全新的 Agent，独立上下文，只读工具集，从头审查。

这消灭了自我评价偏差——AI 问自己"做得好不好"，答案永远是"好"。换一个 AI 来问，答案就诚实了。

审查不需要等人。但质量不打折。

---

### 第四层：自学习 — 同一个坑不踩第二次

`crystal-learn` Skill 自动从失败中提取"不变量"。

比如 AI 改了后端接口但忘了改前端调用——这个失败模式被提取为规则：**"改契约必须 grep 所有消费者"**。

这条规则不是写在文档里等人记住。它被机械化注入到执行层 Skill 中，下次同类操作时**自动生效**。

犯一次错，永久免疫。

---

### 第五层：上下文工程 — AI 不迷路

<!-- [图 ④：Session 隔离] -->

AI 在长对话中最容易犯的错：忘了自己在做什么。

我们的对策：

- **上下文路由**：编辑文件时，自动注入该文件所属领域的规则（bridge 规则、部署规则、场景规则……），不是一次性全加载，是按需精确注入
- **Session 隔离**：多个 AI 并行工作时，每个 Session 通过独立 transcript 文件隔离作用域，互不污染
- **PreCompact 保护**：上下文压缩前，自动保存关键信息，压缩后不丢目标

AI 始终知道自己在做什么。因为有人（hook）在每个关键时刻提醒它。

---

### 第六层：物理隔离 — 不是"请不要"，是"不能"

<!-- [图 ⑤：Schema 级审查隔离] -->

这是整个体系最关键的设计决策。

告诉审查 Agent "不要修改文件"——遵从率约 **70%**。

把 Edit / Write / Bash 从它的工具清单里删掉——遵从率 **100%**。

因为它物理上调不了不存在的工具。

这叫 **Schema 级隔离**。不是靠提示词约束行为，是靠工具清单约束能力。

这个原则贯穿整个 wow-harness：

> **凡是重要的事，不靠说，靠执行。**

## 05 六个月生产验证

wow-harness 不是学术论文。它是从 6 个月的真实生产中**被逼出来的**。

每一条 hook、每一个 gate、每一种隔离模式，都是因为 AI agent 找到了创造性的方法来绕过上一条规则，然后我们才加上的。

举个例子：

我们的 Stop hook 经历了**三轮修复**：

1. **第一轮**：AI 在纯聊天时也弹出完成检查清单 → 加了"是否有写操作"判断
2. **第二轮**：AI 编辑完文件 commit 了再聊天，还是触发 → 改为"编辑文件 ∩ git 未提交变更"交集
3. **第三轮**：两个 AI 并行工作，共享状态互相污染 → 改为基于独立 transcript 的 session 隔离

三轮。一个 Stop hook。

这就是为什么这套系统有效——它不是设计出来的完美系统，是**被现实打磨出来的**。

---

## 06 人只做三件事

装上 wow-harness 之后，人在 AI 开发中的角色变成了三件事：

1. **战略决策** — 做什么
2. **方向纠偏** — 不是这样
3. **最终确认** — PR accept

所有中间环节——写代码、跑测试、做审查、写文档、拆任务、检测漂移——都由 AI 在治理框架内自主完成。

不是因为 AI 足够聪明到不需要管。

是因为治理体系让它**即使犯错也会被拦住**。

## 07 三分钟装好

```bash
git clone https://github.com/NatureBlueee/wow-harness.git
cd wow-harness
python3 scripts/install/phase2_auto.py /path/to/your/project --tier drop-in
```

三个层级，按你的信任度选：

| 层级 | 做了什么 | 适合谁 |
|------|---------|--------|
| **drop-in** | 原样安装，不读你的代码 | 先试试看 |
| **adapt** | 读你的 README + 文档，适配 skill | 正式项目 |
| **mine** | 读你的工作记录，深度适配 | 长期使用 |

安装器是幂等的——跑两次，结果一样。不会覆盖你已有的 CLAUDE.md。

需要：Claude Code CLI + Python 3.9+ + Git。

## 08 和"写更好的 CLAUDE.md"有什么不同

| | 更好的 CLAUDE.md | wow-harness |
|---|---|---|
| 约束方式 | 文本指令 | 机械化 Hook |
| 遵从率 | ~20% | 100% |
| 审查 | 自己查自己 | 独立 Agent，schema 隔离 |
| 并行 | 互相污染 | session-scoped 隔离 |
| 失败模式 | 静默跳过 | 阻断 + 反馈 |
| 学习 | 下次还犯 | 提取不变量，机械注入 |

CLAUDE.md 仍然有用。wow-harness 不替代它——它**执行**它。
