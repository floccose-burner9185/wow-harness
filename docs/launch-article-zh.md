# 我们解决了 AI 开发的六个问题

> wow-harness：一行命令给你的 Claude Code 项目装上完整的工程治理。开源，MIT。

---

你让 AI 帮你写代码。它很聪明，写得快，理解力强。

但用着用着你发现，AI 写代码是一回事，**能不能放心交给它**是另一回事。

以下六个问题，用过 Claude Code 的人或多或少都遇到过。

wow-harness 解决的就是这六个。

---

## 问题一：规则写了，AI 不听

你在 CLAUDE.md 里认真写了一堆规则："提交前跑测试""不要碰生产数据库""改 API 要同步改前端"。

AI 看了。有时候听，有时候不听。

不是故意不听——是**记忆有限**。对话长了，早期的规则会被压缩掉。这是 LLM 的结构性问题，不是你提示词写得不够好。

我们测下来，CLAUDE.md 的规则遵从率大概 **20%**。

### 我们的解法：Hook——规则变成机械装置

<!-- [图 ①] -->

Hook 是 Claude Code 的生命周期钩子。不是文本指令，是**代码**。

AI 要往生产服务器部署？Hook 直接拦住，exit 2，物理阻断。
AI 编辑了文件？Hook 自动注入这个文件所属领域的规则。
AI 要退出？Hook 先检查有没有未提交的代码。

不是"提醒你别做"。是红绿灯——不管你记不记得交规，红灯就是停。

**执行率 100%。**

wow-harness 给你装好 16 个 Hook，覆盖 7 个生命周期阶段。装完自动生效，不需要配置。

---

## 问题二：AI 查不出自己的错

你让 AI 审查自己刚写的代码。它说"没问题，质量很好"。

当然了——**问自己"做得好不好"，答案永远是"好"**。这叫自我评价偏差，是 LLM 的已知问题。

更糟的是：你让审查 AI "只看不改"，它有 30% 的概率会忍不住顺手把代码改了。

### 我们的解法：独立审查 + 工具清单隔离

<!-- [图 ③] -->

wow-harness 有 8 道质量门禁。其中 4 道是**审查门**——自动启动一个全新的 AI 做独立审查。

这个审查 AI 有三个特点：

1. **独立上下文**——不共享写代码那个 AI 的对话记录，从头看
2. **只读工具集**——它的工具清单里没有 Edit、Write、Bash
3. **物理隔离**——不是"请不要改"，是**工具清单里不存在这些工具，调不了**

就像让质检员进车间但不给他扳手——他能看，能报告，但物理上改不了产品。

<!-- [图 ⑤] -->

| 方法 | 遵从率 |
|------|--------|
| 提示词："请不要修改文件" | ~70% |
| 工具清单删掉 Edit/Write | **100%** |

**审查不需要等人。但质量不打折。**

## 问题三：AI 会忘记自己在做什么

一个 session 聊了 50 轮之后，AI 开始漂移——修着 bug A，忘了自己在修什么，开始重构 B。

这不是偶发的，是**上下文压缩的必然结果**。对话越长，早期目标越模糊。

### 我们的解法：按需注入，而不是一股脑塞

<!-- [图 ②] -->

wow-harness 不在 session 开始时把所有规则全加载。太多了 AI 也记不住。

而是**编辑哪个文件，就加载哪个文件的规则**。

编辑部署脚本？自动注入部署安全规则。编辑后端路由？自动注入路由规范。编辑 Bridge 模块？自动注入 Bridge 的五条宪法。

17 套上下文片段，按文件路径精确路由。AI 在编辑的那一刻收到最相关的规则——不多，不少，刚好够用。

加上上下文压缩前自动保存关键信息，AI 始终知道自己在做什么。

## 问题四：同样的坑反复踩

AI 改了后端 API 接口，忘了改前端调用。你修了。

下一次，它又忘了。

因为 LLM **没有跨 session 的记忆**。上次踩的坑，下次还会踩。

### 我们的解法：踩一次坑，永久免疫

wow-harness 有一个叫 `crystal-learn` 的 Skill。它自动从失败中提取**不变量**——就是"这类事情不管什么情况都必须遵守的规则"。

上面那个例子会被提取为："**改契约必须 grep 所有消费者**"。

这条规则不是写在文档里等 AI 记住（遵从率 20%，记得吗？）。它被**机械化注入到执行层 Skill 中**，下次 AI 改 API 时自动生效。

犯一次错。永久免疫。

## 问题五：多个 AI 并行工作会互相干扰

你开了两个 Claude Code 窗口，一个改前端，一个改后端。

结果前端那个 session 要退出的时候，Stop hook 检测到后端 session 的未提交文件，误触发了完成检查。

因为底层状态是**全局共享的**——两个 session 看到的是同一份 git 工作区。

### 我们的解法：每个 session 有自己的视野

<!-- [图 ④] -->

wow-harness 的 Stop hook 不看全局状态。它看**本 session 的 transcript**。

每个 Claude Code session 有一个独立的 `.jsonl` 文件，记录了这个 session 做过的所有操作。Stop hook 只解析这个文件，提取这个 session 写过的文件列表，再和 git 未提交变更取交集。

Session A 的编辑不会影响 Session B 的判定。零共享可变状态。

这个机制经历了三轮迭代才做对（第一轮和第二轮都有漏洞被 AI 的行为模式绕过了）。但现在它覆盖了所有场景。

## 问题六：不是一个 AI 就能干好所有事

让一个 AI 同时做架构设计、写代码、跑测试、做审查、修 bug、写文档——它不是不能做，是**做不好**。

就像让一个人同时当架构师、开发、测试、审查员和运维。能力不是问题，注意力是问题。

### 我们的解法：16 个岗位，各司其职

wow-harness 有 16 个 Skill，对应 16 个专业角色：

- `lead` 管流程（8-Gate 状态机）
- `arch` 做架构
- `guardian-fixer` 修 bug（带 8 关验证）
- `crystal-learn` 从失败中学习
- `bug-triage` 做分诊
- `harness-eng` / `harness-eng-test` / `harness-ops` 做日常开发、测试、运维
- `skill-discovery` 从工作模式中发现新能力
- ……还有 7 个

每个 Skill 有明确的输入输出契约，互不越界。就像一家公司——不是一个全能选手，是一个**分工明确的团队**。

每个 Skill 都有 `{{PLACEHOLDER}}` 槽位，安装时可以填入你项目的上下文，让它适配你的领域。

## 整体来看：开发者得到了什么保障

装上 wow-harness 之后，你的 Claude Code 项目自动获得：

| 保障 | 机制 | 效果 |
|------|------|------|
| **规则一定被执行** | 16 个 Hook，100% 执行率 | 不依赖 AI"记住"规则 |
| **审查一定独立** | 8-Gate + Schema 隔离 | 不可能自审自查 |
| **上下文一定相关** | 17 套片段按路径路由 | 编辑什么文件就看什么规则 |
| **失败一定被学习** | crystal-learn 不变量注入 | 同一个坑不踩第二次 |
| **并行一定隔离** | transcript-scoped Stop | 多 session 互不干扰 |
| **分工一定清晰** | 16 个 Skill 各司其职 | 不是一个 AI 干所有事 |

人只做三件事：

1. **定方向** — 做什么
2. **纠偏** — 不是这样
3. **确认** — PR accept

中间所有环节——写代码、跑测试、做审查、检测漂移——全自动。

## 一行命令，装完就生效

```bash
git clone https://github.com/NatureBlueee/wow-harness.git
cd wow-harness
python3 scripts/install/phase2_auto.py /path/to/your/project --tier drop-in
```

装完你的项目里多了：
- **16 个 Hook** — 自动挂载，session 一开始就生效
- **16 个 Skill** — 随时可用的专业角色
- **15 个自动检查** — API 类型、文档、安全……
- **路径规则** — 编辑文件时自动加载
- **CLAUDE.md** — 自动生成的治理指南

不需要配置。不需要改代码。**开箱即用。**

三个层级，按信任度选：

| 层级 | 做了什么 | 适合谁 |
|------|---------|--------|
| **drop-in** | 原样安装 | 先试试 |
| **adapt** | 读你的文档，适配 Skill | 正式项目 |
| **mine** | 读你的工作记录，深度适配 | 长期使用 |
