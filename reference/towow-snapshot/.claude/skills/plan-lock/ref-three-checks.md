# 接口/字段/时间三检查（PLAN-033 教训沉淀）

plan-lock Step 2.5 强制执行。每当计划中出现接口变更、新增字段或时间相关设计时，必须完成对应检查。

---

## 检查 A：改接口（触发词：重写/替换/新增参数/返回类型变更）

对每个接口变更，grep 所有消费方，逐个回答：

1. **签名兼容吗？** — 参数名、参数数量、返回类型
2. **语义兼容吗？** — 旧调用方在新语义下行为还正确吗？
   - "兼容层"不是免审金牌。兼容层的**语义**可能在新模型下是错的
   - 例：`refund_one()` 签名不变，但新模型 run 预扣了 20，兼容层只退 1 → 19 个 invocation 泄漏
3. **新参数需要调用方提供吗？** — 如果有新参数且默认值在某些路径下不安全，必须在计划中显式列出

**输出**：消费方影响矩阵表（消费方 × 签名兼容 × 语义兼容 × 需适配）

---

## 检查 B：加字段（触发词：新增列/新增字段/新增参数/新增返回字段）

对每个新字段，追踪 8 条路径：

| # | 路径 | 检查 |
|---|------|------|
| 1 | 创建/初始化 | 在哪里创建？默认值是什么？默认值在所有场景安全吗？ |
| 2 | 写入 | 哪些代码路径会设置它？有没有路径漏设？ |
| 3 | 读取 | 哪些代码路径使用它？读到默认值时行为正确吗？ |
| 4 | 传递 | 作为参数传递的链条完整吗？中间环节有没有漏传？ |
| 5 | 缺失/默认 | 旧记录没有这个字段 → 默认值在所有消费路径下都安全吗？ |
| 6 | 旧数据迁移 | ALTER TABLE 有吗？迁移后旧记录的值正确吗？ |
| 7 | 跨时间边界 | 刷新/过期/跨日时这个字段的状态正确吗？ |
| 8 | 并发 | 两个操作同时读写这个字段安全吗？ |

**输出**：字段生命周期矩阵（字段 × 8 条路径 × 状态）

---

## 检查 C：涉及时间（触发词：刷新/过期/超时/定时/每日/cron）

对每个时间相关设计，检查 3 个边界：

1. **恰好在边界时刻** — 恰好在刷新/过期那一秒会怎样？
2. **跨越边界** — 操作开始于边界前、结束于边界后会怎样？
   - 例：23:50 预扣 → 00:05 结算 → 跨日刷新已清零 reserved_today → 结算增加了 used_today → 当日额度被污染
3. **并发在边界附近** — 两个操作都在边界前后几秒内执行会怎样？

**输出**：时间边界场景表（边界 × 3 种检查 × 结果）

## 真实案例（PLAN-033 五轮审查）

| 轮次 | 发现 | 对应检查 | 如果第一轮就做了检查 |
|------|------|---------|-------------------|
| R2 | scheduler refund_one 只退 1 而非 20 | A-语义兼容 | R1 就能发现 |
| R3 | coordinator _fail_run 同样问题 | A-语义兼容 | R1 就能发现 |
| R2 | settle() 漏传 bonus_reserved | B-传递链 | R1 就能发现 |
| R5 | legacy demand quota_reserved=0 | B-缺失/默认 | R1 就能发现 |
| R3 | 跨日结算污染当日额度 | C-跨越边界 | R1 就能发现 |
| R1 | queued run 预留永不释放 | C-恰好在边界 | R1 发现了 ✅ |
