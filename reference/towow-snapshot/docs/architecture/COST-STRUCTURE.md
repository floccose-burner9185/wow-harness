# Discovery + Collaboration 成本结构

**最后验证**: 2026-03-23, 基于代码实际链路追踪

## Discovery 阶段

### LLM 调用（有成本）

| 环节 | 触发时机 | 次数 | 模型 | 可省？ |
|------|----------|------|------|--------|
| D1-D5 描述生成 | Agent 注册/profile 文本变更 | 每个 agent 1 次 | scene config 指定 | 否 — 编码基础 |
| D1-D5 描述生成 | 用户发需求时 | 每次需求 1 次 | scene config 指定 | 否 — 匹配基础 |

**去重机制**: `profile_hash` — profile 文本不变则跳过重新编码（`service.py:140`）。服务重启从快照恢复向量，不触发 LLM。

### 向量/数学运算（成本 ≈ 0）

| 环节 | 说明 |
|------|------|
| bge-m3 embedding | D1-D5 文本 → 5 维向量，CPU 本地推理 |
| 6 算子打分 (OP1-OP6) | 余弦相似度、互补匹配等纯数学运算，对每个候选人并行执行 |
| RRF 融合排序 | 加权排序，纯算术 |

**关键认知**: OP1-OP6 是确定性向量算子，不是 LLM 调用。减少算子数量不省钱，只降低匹配质量。OP7 (LLM Hypothesis) 存在但未在生产启用。

## Formulation 阶段

| 环节 | 次数 | 可省？ |
|------|------|--------|
| 需求澄清多轮对话 | 2-5 轮 LLM 调用 | 否 — 产品核心体验 |

## Crystallization 阶段（成本大头）

| 环节 | 次数 | 可省？ |
|------|------|--------|
| 多 agent 多轮协商 | k 个参与者 x N 轮 | 否 — 产品核心价值 |

这是不可压缩的成本 — 协作质量和 LLM 调用量正相关。

## 成本杠杆

| 杠杆 | 效果 | 状态 |
|------|------|------|
| profile hash 去重 | 避免重复编码 | 已实现 |
| 快照恢复 | 重启不触发全量重编码 | 已实现 (PLAN-039) |
| BYOK | 用户自带 API key | 已实现 |
| 模型降级 | D1-D5 生成用更便宜模型 | 可选优化，未做 |
| k 值控制 | 减少协商参与者数 | 已暴露给用户 |

## 参数说明

| 参数 | 含义 | 默认 | 范围 |
|------|------|------|------|
| `k` | 协商参与者数量 | 3 | 1-50 |
| `max_candidates` | matching 返回候选人数 | 10 | 1-50 |
| 算子数量 | OP1-OP6 固定 6 个 | 6 | 不可配置（无需配置，零成本） |
